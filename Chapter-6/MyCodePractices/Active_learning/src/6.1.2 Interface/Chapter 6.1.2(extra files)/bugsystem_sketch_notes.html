Feynman Sketch Notes • হাতে আঁকা বোঝার মতো • Chapter 6.1.2

1
Interface Variable new করা যায় না — কিন্তু variable বানানো যায়
Livable l = new Bee("Bumble", 1, 0.1, "Garden");
Livable হলো ফর্মের ঘর — খালি জায়গা। new Bee() হলো যে সেই ঘরে বসলো। ফর্মকে নিজেই চেয়ারে বসাতে পারবে না।
2
instanceof check Airport security — wings আছে কিনা চেক করো
if (b instanceof Flyable) { b.fly(); }
প্রতিটা bug কে জিজ্ঞেস করো — Flyable আছো? হ্যাঁ বললে fly() call করো। না বললে skip।
3
Polymorphism একই variable — আলাদা আলাদা object
Flyable f = new Butterfly(...);
f = new Bee(...);  // same variable, different behavior!
একটাই বাক্স Flyable f — কিন্তু ভেতরে যে কেউ ঢুকতে পারে। Butterfly ঢুকলে Butterfly এর fly(), Bee ঢুকলে Bee এর fly()।
4
Constant check সরকারি নিয়ম — কেউ বদলাতে পারবে না
if (alt > Flyable.MAX_ALT) { "Too High!" }
MAX_ALT = 500 মানে interface এ লেখা সরকারি নিয়ম। Java এটাকে public static final বানিয়ে দেয়। Bee ও বদলাতে পারবে না, Butterfly ও পারবে না।
5
Multiple implements WaterBeetle — দুই দলের সদস্য একসাথে!
WaterBeetle wb = new WaterBeetle(...);
wb.crawl(5);  // Crawlable থেকে
wb.swim(2);  // Swimmable থেকে
Ant শুধু হাঁটতে পারে। Fish শুধু সাঁতার পারে। কিন্তু WaterBeetle দুটোই পারে — কারণ সে দুটো interface implement করে। Pizza Project এ GrilledPizza এর মতো — bake + grill।
6
Sealed Interface VIP Club — শুধু দুজন member!
BugBehavior bb = new Bee("Bumble", 1, 0.1, "Garden");
sealed মানে বন্ধ দরজা। permits Livable, Bug — এই দুজনের নাম লেখা। বাইরের কেউ extend/implement করতে এলে Java বলবে: না!
7
Comparable + Generic দাঁড়িপাল্লা — কে ভারী?
processBug(bee1, butterfly1);
<T extends Comparable<T>> void processBug(T a, T b)
compareTo() দাঁড়িপাল্লার মতো। <T extends Comparable<T>> মানে — T যাই হোক, সে compare করতে পারে। Bee দাও, Butterfly দাও — সব চলবে।
8
Pattern Matching instanceof পুরনো vs নতুন — একলাইনে দেখো আর ধরো
if (item instanceof Flyable f) { f.fly(100); }
পুরনোতে দুটো কাজ আলাদা — আগে check করো, তারপর cast করো। নতুনতে instanceof Flyable f — check আর capture একসাথে। f সাথে সাথে ready, আলাদা cast দরকার নেই।
